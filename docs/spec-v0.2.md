# setnote — DJセットリスト共有サービス 仕様書 (v0.2)

**サービス名: setnote**(セットの記録の意。ドメイン: `setnote.yu-web.site`)

## 1. コンセプト

DJが自分のセットリストを作成・管理し、SNS(主にTwitter/X)で共有するためのサービス。
構造は3層:

1. **管理側** — ログインしてセットリストを作成・編集・公開管理する画面
2. **閲覧側** — アカウント不要で誰でも見られる公開ページ(楽曲埋め込み・OGP対応)
3. **分析側** — 曲の使用回数やリンククリックを見られるオマケ機能

ユーザーページ(DJごとの一覧ページ)は作らず、各セットリストは独立したURLとしてSNSに貼る使い方を想定。ユーザーページは将来拡張として追加可能な設計にしておく。

---

## 2. URL設計

| 用途 | URL |
|---|---|
| 公開セットリスト | `setnote.yu-web.site/s/{nanoid}` |
| 固定ページ | `/login`, `/terms`, `/privacy` など |
| 管理画面 | `/admin` 配下(または別サブドメイン) |

- `{nanoid}` はセットリスト作成時に発行する固定長(8〜10文字)のランダムID
- セットリスト名を変更してもURLは不変
- 非公開化 → 再公開しても同じURL
- `/s/` プレフィックスにより固定ページとの名前空間を分離(ルーティング・CDNキャッシュ・ログ集計が単純になる)
- URLにユーザー識別子は含めない。セットリストとユーザーの紐付けはDB側で管理

## 3. アカウント・認証

- 認証基盤は **Amazon Cognito**。まず Google 連携から開始
- Twitter(X) ログインは API の料金・規約の状況を見て IdP として後追い追加
- アカウントの一意キーは**メールアドレス**(内部利用のみ、公開情報には一切出さない)
- サインアップ時に**ユーザー名(表示名)の登録を必須**とする。将来のユーザーページ等で利用

## 4. セットリストのデータ構造

### セットリスト

| 項目 | 必須 | 備考 |
|---|---|---|
| セットリスト名 | ✅ | |
| イベント名 | – | |
| イベントリンク | – | |
| 実施日 | – | |
| 曲カード(順序付きリスト) | – | 下記 |

### 曲カード

| 項目 | 必須 | 備考 |
|---|---|---|
| 楽曲名 | ✅ | 名無しの曲もスペース等で入力してもらう(YouTube等の既存慣習に合わせる)。オートコンプリートの同一曲判定にも必要 |
| 作者 | – | |
| 楽曲リンク | – | YouTube / Spotify / SoundCloud などすべて許可 |
| 入手元 | – | リンクでも自由文でもOK。URLチェックは行わない |
| カスタム項目 | – | key-value形式。BPM・キー・コメント等を自由に追加 |

### 曲カードのデータ構造(実装)

```ts
type CustomField = { id: string; label: string; value: string };
type Track = {
  id: string;                  // クライアント発行(並べ替え・React key用, crypto.randomUUID)
  title: string;               // 楽曲名(必須)
  artist: string;              // 作者
  songLink: string;            // 楽曲リンク
  source: string;              // 入手元(URL/自由文)
  customFields: CustomField[]; // カスタム項目(BPM/Key/コメント等)
};
```

### 曲データの方針

- 曲マスタ(一括編集)は**作らない**
- 過去に自分が入力した曲を入力途中でサジェスト表示し、選択するとリンク等含め全項目をコピー入力(オートコンプリート)
- 使用回数カウントは同一曲判定に依存するが、あくまでオマケ精度でよい

## 5. 状態管理(公開/非公開)

- セットリストは常に**単一の最新版**を保持する(公開スナップショットは持たない)
- **「公開」= 表示可能にする / 「非公開」= 表示不可(下書きに戻す)** の制御だけを行う。保存した内容は**即座に公開ページへ反映**される(再公開は不要)
- 状態: `下書き(非公開)` / `公開中`(旧「公開取り消し(unpublished)」は下書きに統合)
- 非公開・削除・存在しないIDへのアクセスは**共通404ページ**へ(存在を区別しない。真っ白なページにはしない)
- 保存時に完了トーストを表示。公開時に公開URLを提示

> **実装状況**: `PUT /:id`(保存=最新版) / `POST|DELETE /:id/publish`(公開=表示ON/OFF)実装済み。公開ページ(`GET /:id`)は常に最新の公開データを返す。編集画面の保存・公開UIも実装済み。自動保存(#35)は見送り。

## 6. 編集画面(管理側)

- ログイン → アカウントごとの管理画面 → 「セットリストを新規作成」で編集画面へ遷移
- 曲はカード形式で追加。途中のカードの削除、**ドラッグ&ドロップでの並び替え**に対応
- **スマホ・PC両対応**(モバイルでのドラッグ操作を考慮したUI)
- 楽曲名の入力途中に、過去入力からのサジェストを入力欄の下に表示

## 7. 公開ページ(閲覧側)

- アカウント登録不要で閲覧可能
- 楽曲リンクは**ファサード方式の埋め込み**:
  - 初期表示はサムネイル+再生ボタン風の自前UI(ページを軽く保つ)
  - タップでクリック計測を送信しつつ iframe(YouTube/Spotify/SoundCloud埋め込み)に差し替え → その場で再生
- 入手元がURLの場合はリンクとして表示(クリック計測対象)、自由文はそのまま表示
- 注意点: Spotify埋め込みは未ログイン閲覧者には30秒プレビューになる

## 8. OGP

- リンクをTwitter等に貼るとセットリスト名・イベント名が表示される
- 可能ならセットリスト名+イベント名入りの**動的OG画像**を生成

## 9. 分析画面(管理側・オマケ機能)

- ある曲を自分のセットリストで**何回使ったか**
- 曲ごとの**楽曲リンククリック数**・**入手元クリック数**
- セットリストのアクセス数(PV)を分母にした**コンバージョン率**(セットリスト間の比較用)
- 分母の定義(UU/ボット除外/自己アクセス除外)は初期は雑でよい。精緻化は後々のアップデート

## 10. その他

- 利用規約・プライバシーポリシーページを用意(クリック計測を行うため特に必要)
- 404は全ケース共通の専用デザインページ

## 12. 決定事項 / 残課題(2026-07-11 更新)

### 決定済み

- **技術スタック**: React + Vite + TypeScript(フロント) / Hono + TypeScript + DynamoDB(Lambda) / Terraform(S3+CloudFront+API Gateway+Lambda) / 認証は Amazon Cognito。UI は **shadcn/ui**(Radix + Tailwind v4, neutral テーマ, ダーク固定)
- **MVPの切り方**: 「作成 → 編集 → 公開 → 閲覧」のコア動線を優先。オートコンプリート・分析は後追い
- **nanoid**: 10文字(`nanoid(10)`)で確定
- **日付フォーマット**: 実施日は `YYYY-MM-DD`(HTML date input)。一覧表示はタイムゾーン非依存で `YYYY/M/D` に整形
